## 1. 项目目标与成功标准

### 1.1 业务目标

| ID | 描述 | 优先级 |
|----|------|--------|
| G-01 | 用户可登记正在使用的药物，系统在 FDA 发布相关召回时主动通知 | P0 |
| G-02 | 用户可查询/扫描药物，判断是否与已知召回相关（至少 NDC 级，理想为批次级） | P0 |
| G-03 | 按 FDA Class I / II / III 展示召回严重程度 | P0 |
| G-04 | 快速上线 Web MVP，减少重复开发（为后续 App 预留架构） | P0 |
| G-05 | 为药房等 B2B 场景保留扩展空间（本期可不实现） | P2 |

### 1.2 成功指标（待确认数值）

| ID | 指标 | 说明 |
|----|------|------|
| M-01 | 召回同步延迟 | FDA 数据更新后 ≤ X 小时内完成入库与通知 |
| M-02 | NDC 级匹配准确率 | ≥ 99%（召回公告含明确 NDC 时） |
| M-03 | 批次级匹配 | 有批号时误报率 < Y%（需建立测试集） |
| M-04 | 邮件通知送达率 | ≥ Z% |
| M-05 | 首药添加转化率 | 待产品定义 |

### 1.3 明确不做（Non-Goals）— MVP

- 不提供用药建议、停药建议（仅引用 FDA/厂商原文与链接）
- 不做药物相互作用（V2 单独立项）
- 不做化妆品、食品召回（后续版本）
- 不做处方开具、药房 ERP 对接（B2B 单独立项）

## 3. 功能需求

### 3.1 账户与用户（AUTH & PROFILE）

| ID | 需求 | 验收要点 | 优先级 | 阶段 |
|----|------|----------|--------|------|
| AUTH-01 | 用户注册 | 邮箱 + 密码或 Magic Link；可登录登出 | P0 | MVP |
| AUTH-02 | 密码重置 | 邮件重置链接有效 | P0 | MVP |
| AUTH-03 | 用户资料（可选） | 姓名、时区等；扩展字段均可选 | P1 | MVP |
| AUTH-04 | 可选人口统计学字段 | 年龄、种族等：可选、可跳过、可删除；无用途则不收集 | P2 | 商定 |
| AUTH-05 | 账户删除与数据导出 | 支持删除账号与导出个人数据 | P1 | MVP |
| AUTH-06 | 多因素认证 | 提升账户安全 | P2 | V1.1 |

### 3.2 药箱 / 用药登记（MEDICATION CABINET）

| ID | 需求 | 验收要点 | 优先级 | 阶段 |
|----|------|----------|--------|------|
| MED-01 | 手动添加药物 | 按名称搜索并选择 NDC | P0 | MVP |
| MED-02 | 记录 NDC（必填） | 保存标准 NDC | P0 | MVP |
| MED-03 | 可选：生产厂家 / 商品名 | 区分多厂家同名药 | P1 | MVP |
| MED-04 | 可选：批号（Lot）、效期 | 用于批次级匹配 | P1 | MVP |
| MED-05 | 用药监控时长 | 默认 2–3 个月；到期策略见 [OQ-01](#9-待拍板问题-open-questions) | P0 | MVP |
| MED-06 | 编辑 / 删除 / 停用药物 | 状态变更后调整或停止通知 | P0 | MVP |
| MED-07 | 用药历史查看 | 已添加、已停用及时间线 | P1 | MVP |
| MED-08 | 家庭成员 / 多 Profile | 为家人管理独立药箱 | P1 | V1.1 |
| MED-09 | 同成分扩展（RxNorm） | 仿制药 NDC 不同时仍能关联召回 | P1 | V1.1 |

### 3.3 条码扫描（BARCODE）

| ID | 需求 | 验收要点 | 优先级 | 阶段 |
|----|------|----------|--------|------|
| SCAN-01 | 扫描药瓶条码识别 NDC | 解析 GS1/NDC，展示药品信息供确认 | P1 | V1.1 |
| SCAN-02 | 解析批号、效期（若码内含） | 写入 MED-04 | P1 | V1.1 |
| SCAN-03 | 扫码即时召回查询（不加入药箱） | 一次性查询结果页 | P1 | V1.1 |
| SCAN-04 | 扫码失败引导 | 权限、光线、改用手动输入 NDC | P1 | V1.1 |
| SCAN-05 | 原生 App 相机体验 | 优于 Web PWA | P2 | App |

**MVP 替代方案**：搜索 + 手动输入 NDC（不含扫码）。

### 3.4 FDA 数据与召回库（DATA & SYNC）

| ID | 需求 | 验收要点 | 优先级 | 阶段 |
|----|------|----------|--------|------|
| DATA-01 | 从 openFDA Drug Enforcement 拉取召回 | 定时任务；失败可告警 | P0 | MVP |
| DATA-02 | 增量同步 | 仅处理新/变更记录 | P0 | MVP |
| DATA-03 | 存储召回等级 Class I/II/III | 列表与详情可展示 | P0 | MVP |
| DATA-04 | 存储召回元数据 | 原因、厂商、日期、状态、官方链接等 | P0 | MVP |
| DATA-05 | 解析 `code_info` 批号/效期 | 结构化 lot 列表；允许迭代规则 | P1 | MVP |
| DATA-06 | NDC 目录同步 | 支持搜索与 NDC 校验（openFDA NDC / RxNorm） | P0 | MVP |
| DATA-07 | 展示数据更新时间 | 召回详情显示「数据截至」 | P0 | MVP |
| DATA-08 | 多数据源校验（可选） | 降低漏同步风险 | P2 | V1.1 |
| DATA-09 | 化妆品 / 食品数据源 | 独立数据管道 | P2 | V2+ |

**参考**：[openFDA Drug Enforcement](https://open.fda.gov/apis/drug/enforcement/)

### 3.5 匹配引擎（MATCHING）

| ID | 需求 | 验收要点 | 优先级 | 阶段 |
|----|------|----------|--------|------|
| MATCH-01 | NDC 精确匹配 | 用户 NDC 在召回 NDC 列表中则命中 | P0 | MVP |
| MATCH-02 | 批号匹配（有批号时） | 规范化后匹配；无法确认则降级 | P1 | MVP |
| MATCH-03 | 仅 NDC 命中、无批号 | 文案：「可能涉及，请核对瓶身批号」 | P0 | MVP |
| MATCH-04 | 新召回入库触发用户检查 | 非仅定时日报 | P0 | MVP |
| MATCH-05 | 用户新加药触发历史召回检查 | 添加药物时立即扫描历史召回 | P0 | MVP |
| MATCH-06 | 匹配结果去重 | 同一用户同一召回只通知一次 | P0 | MVP |
| MATCH-07 | 同成分 / 多厂家扩展 | 降低仿制药漏报 | P1 | V1.1 |

### 3.6 通知（NOTIFICATIONS）

| ID | 需求 | 验收要点 | 优先级 | 阶段 |
|----|------|----------|--------|------|
| NOTIF-01 | 邮件通知 | 含等级、药名、原因摘要、官方链接 | P0 | MVP |
| NOTIF-02 | 站内通知中心 | 可查看历史告警 | P0 | MVP |
| NOTIF-03 | 按 Class 区分模板 | Class I 更醒目；II/III 引用 FDA 原文说明 | P0 | MVP |
| NOTIF-04 | 用户通知偏好 | 邮件开关、摘要频率等 | P1 | MVP |
| NOTIF-05 | SMS 通知 | Twilio 等；需用户 opt-in | P2 | V1.1+ |
| NOTIF-06 | Push 通知 | 需原生 App | P2 | App |
| NOTIF-07 | 送达与失败重试 | 日志可追踪 | P1 | MVP |

### 3.7 召回查询与展示（RECALL UI）

| ID | 需求 | 验收要点 | 优先级 | 阶段 |
|----|------|----------|--------|------|
| UI-01 | 召回列表 | 可按 Class、日期筛选；是否公开见 [OQ-03](#9-待拍板问题-open-questions) | P1 | MVP |
| UI-02 | 召回详情页 | 完整字段 + FDA 原文链接 | P0 | MVP |
| UI-03 | 「我的药箱受影响」聚合页 | 登录用户专属 | P0 | MVP |
| UI-04 | 扫描/查询结果页 | 命中 / 未命中 / 不确定 三态 | P1 | V1.1 |
| UI-05 | 免责声明 | 非医疗建议、不保证早于 FDA 获知 | P0 | MVP |

### 3.8 管理后台（ADMIN）— 内部

| ID | 需求 | 验收要点 | 优先级 | 阶段 |
|----|------|----------|--------|------|
| ADM-01 | 同步任务监控 | 最近成功时间、错误日志 | P0 | MVP |
| ADM-02 | 手动触发同步 | 全量/增量 | P1 | MVP |
| ADM-03 | 匹配规则 / 黑名单 | 减少已知误报 | P2 | V1.1 |
| ADM-04 | 用户与通知审计日志 | 合规与排查 | P1 | MVP |

### 3.9 第二版：药物相互作用（V2 — 独立 Epic）

| ID | 需求 | 验收要点 | 优先级 | 阶段 |
|----|------|----------|--------|------|
| DDI-01 | 药箱内药物两两冲突检测 | 独立数据源 + 医学免责声明 | P2 | V2 |
| DDI-02 | 冲突严重度与通知 | 可复用通知通道 | P2 | V2 |
| DDI-03 | 法规与产品定性评审 | SaMD / 临床决策支持边界 | P0（流程） | V2 前 |

## 4. 非功能需求

| ID | 类别 | 需求 | 优先级 |
|----|------|------|--------|
| NFR-01 | 可用性 | 核心服务可用性目标（如 99.5%）待运维确认 | P1 |
| NFR-02 | 性能 | 单用户药箱匹配 < 2s；召回列表首屏 < 3s | P1 |
| NFR-03 | 安全 | HTTPS；密码安全哈希；敏感字段加密存储 | P0 |
| NFR-04 | 隐私 | 最小必要收集；隐私政策与用户协议 | P0 |
| NFR-05 | 合规 | 不收集与产品无关信息；支持删除与导出 | P0 |
| NFR-06 | 可访问性 | WCAG 2.1 AA 基础 | P2 |
| NFR-07 | 国际化 | MVP 仅 en-US；架构预留 i18n | P1 |
| NFR-08 | 可观测性 | 日志、指标；同步与通知失败告警 | P0 |
| NFR-09 | 限流 | openFDA API Key 与 rate limit 处理 | P0 |
| NFR-10 | 灾备 | 数据库定期备份 | P1 |

## 6. 合规与法律

| ID | 待确认事项 | 建议 |
|----|------------|------|
| LEG-01 | 是否构成 SaMD / 医疗器材软件 | 定位为信息聚合，不提供治疗建议 |
| LEG-02 | HIPAA 适用性 | 按 PHI 标准设计，即使非 Covered Entity |
| LEG-03 | 隐私政策、ToS、免责声明 | 上线前必须就绪 |
| LEG-04 | 人口统计字段 | 无明确分析用途则 V1 不收集 |
| LEG-05 | 通知文案法律责任 | II/III 类引用 FDA 原文，避免「必须停药」等表述 |

## 10. 用户故事示例

| 故事 | 关联需求 |
|------|----------|
| 作为用户，我想用邮箱注册并添加正在服用的药物（NDC），以便有召回时收到通知 | AUTH-01, MED-01/02, MATCH-05 |
